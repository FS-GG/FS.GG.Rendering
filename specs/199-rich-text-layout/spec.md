# Feature Specification: Symbology Full Rich-Text Layout (paragraph layout + decoration/slant/tracking)

**Feature Branch**: `199-rich-text-layout`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "full rich text layout" — the next **deferred symbology backlog item**. Spec 196 ([label / glyph-text channel](../196-symbology-label-text/spec.md)) shipped a single-line identity label, spec 197 ([multi-line / paragraph label](../197-symbology-multiline-label/spec.md)) widened it to several lines, and spec 198 ([rich-text label runs](../198-symbology-rich-text-label/spec.md)) let the label carry **styled runs** with per-run **colour, weight, and size** — but 198 explicitly deferred (198 FR-018) the two things this feature delivers: **full rich-text layout** (per-paragraph alignment incl. justification, explicit paragraph/line structure within styled runs) and the **typographic run attributes beyond colour/weight/size** (italic / slant, underline, strike-through, letter-spacing / tracking). This feature completes the rich-text label: a label may set **how its lines are aligned within the region** and each run may carry **decoration and slant and tracking** — still tofu-free, still fitted to the region, still an opt-in inspection-detail channel, still **byte-identical** when the label uses none of the new capabilities.

## Background

The symbology label channel has widened in three deliberate steps: 196 added an **optional identity label** (`Token.Label`) sited per grammar (Token / Badge / Ring), fitted with real text measurement, tofu-free at the render edge, and byte-identical to the pre-feature symbol when absent; 197 let the label span **several short lines** (hard breaks + measured soft-wrap), capped and fitted to the region; 198 let the label carry an **ordered sequence of styled runs**, each with its own **colour, weight, and size** (`LabelText.Rich of LabelRun list`, `LabelRun = { Text; Color; Weight; Scale }`), measured **per run** and flowed into the region.

Two limits remain after 198, and they are exactly what was deferred in 198 FR-018:

1. **No paragraph layout.** Styled runs only *flow* into the region; they cannot be **aligned** (the block is sited but lines are not leading/centre/trailing aligned within the region, and there is no **justification** that distributes space to fill the width). A `Rich` label also has no first-class **paragraph / line structure** — multi-line emphasis must be faked through run text, with no per-paragraph alignment.

2. **No typographic decoration or slant or tracking.** A run can change colour, weight, and size, but not **slant (italic)**, **underline**, **strike-through**, or **letter-spacing (tracking)** — the attributes that make rich text "full" and let a run read as quoted, deleted, linked-style, or visually distinct without competing for the weight/colour budget.

This feature lifts both limits as a **bounded completion** of the existing label channel, not a new subsystem. Designers running the `fs-gg-symbology` render→eyeball→tweak loop gain: per-paragraph **alignment** (leading / centre / trailing / justify) within the label region; explicit **paragraph and line breaks** inside a styled (`Rich`) label; and per-run **italic / underline / strike-through / letter-spacing**. Everything stays measured per run in that run's own style, fitted (wrap / cap / truncate) so the block never overflows the footprint or collides with the sigil / health / other channels, deterministic under a fixed measurement provider, and tofu-free at the render edge. It remains **opt-in and additive**: a label that uses none of the new capabilities renders **byte-identically** to spec 198 (and through 198's own zero-drift chain, a plain label is byte-identical to 197 and a no-label symbol to the pre-feature symbol).

What stays **out of scope** (still deferred, FR-019): inline images, hyperlinks, bullet / numbered **lists**, **per-glyph** styling, automatic generation or styling of labels from stats without a human in the loop, label-bound **motion** / animating runs, advanced bidirectional / complex-script typography beyond what the installed measurer already supports, any new **GPU/compute** path, and shipping **new font files**.

## User Scenarios & Testing *(mandatory)*

A designer has a roster whose identity labels need real document structure — a centred callsign on its own line, a justified descriptor block beneath it, a *quoted* status in italics, a struck-through retired code — but spec 198 can only stack styled runs flush-flowed with no alignment and no decoration. This feature lets the same `'stats -> Token` mapping carry a label that sets **per-paragraph alignment** (leading / centre / trailing / justify), that breaks into **explicit paragraphs and lines** inside a styled label, and whose runs add **italic, underline, strike-through, and letter-spacing** to the existing colour / weight / size — all drawn with real glyphs (never tofu), measured per run, fitted so the block never overflows or clips, in every grammar (Token, Badge, Ring). A label that uses **none** of the new capabilities renders **byte-identically** to spec 198; a plain label is byte-identical to 197; a no-label symbol is byte-identical to the pre-feature symbol — the additions are purely additive and opt-in.

### User Story 1 - Full per-run typography (italic / underline / strike / tracking) (Priority: P1)

A designer attaches a styled label whose runs add **typographic decoration and slant and tracking** beyond colour/weight/size — for example an *italic* quoted status, an underlined tag, a struck-through retired code, a letter-spaced acronym — to a unit's encoded state and renders the symbol. Each run is drawn in reading order with its **own slant, underline, strike-through, and letter-spacing** (in addition to its colour / weight / size), each with **real glyphs (not tofu)**, in every grammar. A run that sets **none** of the new attributes renders **byte-identically** to spec 198, so a 198-era label is unchanged — the new attributes are purely additive and opt-in.

**Why this priority**: It is the core "full rich text" payload — the decoration / slant / tracking vocabulary that lets a run read as quoted, deleted, or distinct without spending the colour/weight/size budget — and the smallest independently demonstrable slice. Shipping it alone closes the larger half of the deferred 198 FR-018 typography gap.

**Independent Test**: Build `Token`s carrying (a) no label, (b) a 198-era styled label with no new attributes, and (c) a label whose runs set italic / underline / strike-through / letter-spacing; render each in each grammar through the headless render bridge, and confirm (1) the new-attribute label's rasterised output shows each run drawn with its decoration / slant / tracking and real glyphs (non-tofu); (2) a label using no new attributes is byte-identical to spec 198, and a no-label scene is byte-identical to the pre-feature symbol; and (3) two `Token`s whose labels carry the **same characters** but **different decoration/slant/tracking** produce observably different rendered output (each is a channel that changes the bytes).

**Acceptance Scenarios**:

1. **Given** a `Token` whose label has runs setting italic, underline, strike-through, or letter-spacing, **When** it is rendered in any grammar through the render bridge, **Then** each run is drawn in reading order with its decoration / slant / tracking and real glyphs (tofu-free) and legible at the target on-board size.
2. **Given** two `Token`s identical except one run sets a new typographic attribute (e.g. italic) and the other does not, **When** both are rendered, **Then** their scenes differ in a way attributable to that attribute, and neither raises.
3. **Given** a `Token` whose label sets **no** new typographic attributes (a 198-era styled label, or a plain or absent label), **When** it is rendered, **Then** its scene is byte-identical to spec 198 / 197 / the pre-feature symbol respectively (layered zero drift).
4. **Given** the same new-attribute label rendered twice (in-process and across processes) under the same text-measurement provider, **When** the two scenes are compared, **Then** they are byte-identical (determinism).

### User Story 2 - Paragraph layout: alignment, justification, and explicit breaks (Priority: P2)

A designer gives a label real document layout: the lines are **aligned** within the symbol's label region — leading, centre, trailing, or **justified** (space distributed to fill the region width, the last line of a paragraph left un-justified) — and a styled (`Rich`) label is broken into **explicit paragraphs and lines** rather than one flush-flowed block. Alignment and justification operate **inside** the existing per-grammar label region and are **fitted** with real per-run measurement: lines still wrap at measured boundaries, the drawn line count is still **capped** to the region, surplus still degrades gracefully (truncation / ellipsis on the last drawn line), each line's height still follows its tallest run on a shared baseline, and the block never overflows the footprint or overlaps the sigil / health / other channels. A label with **default** alignment and no explicit breaks lays out **byte-identically** to spec 198 (default = the existing flow).

**Why this priority**: Alignment / justification / paragraph structure is the "layout" half of "full rich text layout", and the half that needs the per-run fit machinery to stay trustworthy — a justified or centred block that overflows into the sigil/health channels is worse than 198's flow. It builds on the run typography proven by P1, so it follows.

**Independent Test**: Render `Token`s whose labels set each alignment (leading / centre / trailing / justify) and explicit paragraph/line breaks, including content wider/taller/more-numerous than the region holds; confirm (1) each alignment places the drawn lines correctly within the per-grammar region (centre is centred, trailing is right-sited, justify fills the width with the last paragraph line un-justified); (2) the block stays within the footprint — measured per run — with no mid-glyph clip, capped line count, surplus truncated rather than overflowing, mixed-size lines sized to their tallest run; (3) the block does not overlap an adjacent channel; and (4) a label with default alignment and no explicit breaks is byte-identical to spec 198.

**Acceptance Scenarios**:

1. **Given** a label that sets centre (or trailing) alignment, **When** the symbol is rendered, **Then** the drawn lines are centred (or trailing-sited) within the per-grammar label region and the block stays inside the footprint.
2. **Given** a label that sets justified alignment over content that wraps to multiple lines, **When** the symbol is rendered, **Then** inter-word space is distributed so each wrapped line fills the region width while the last line of each paragraph is left un-justified, and no run is clipped mid-glyph.
3. **Given** a styled (`Rich`) label with explicit paragraph and line breaks, **When** the symbol is rendered, **Then** the breaks produce the authored line/paragraph structure (paragraphs may carry their own alignment) within the region, capped and fitted as in 197/198.
4. **Given** a label whose aligned/justified content exceeds the region, **When** the symbol is rendered, **Then** the drawn line count is capped and the surplus is indicated (e.g. ellipsis on the last drawn line) without overflowing into adjacent channels.
5. **Given** a label with **default** alignment and no explicit breaks, **When** the symbol is rendered, **Then** its layout is byte-identical to spec 198 (default alignment reproduces the existing flow).
6. **Given** a degenerate `Token` (`R <= 0`) that also carries an aligned/decorated label, **When** it is rendered, **Then** the existing visible placeholder is produced and no exception is raised (the placeholder rule wins over the label).

### User Story 3 - Laid-out labels on review boards, governed unchanged (Priority: P3)

A designer assembles a review board (gallery / motion filmstrip) of a fully-laid-out-label roster in any grammar to A/B legibility, and runs the existing legibility linter (spec 194). Aligned / decorated labels render reproducibly on the board; the linter continues to return a **grammar-independent** verdict and treats the label — however laid out or decorated — as an **inspection-detail** channel (it may report label usage but does not change its governance of the pre-attentive channels). The design-loop skill documents full rich-text layout with its legibility caveats: keep paragraphs short, use a restrained set of alignments and decorations, don't let underline/strike/italic or justification crowd the region or impersonate the faction/state pre-attentive encodings, inspection-only, requires the real measurer to be tofu-free, complements the vector sigil.

**Why this priority**: It turns laid-out labels into something the loop can compare and govern, and confirms the linter and skill keep working. Strictly additive on top of P1/P2.

**Independent Test**: Render a fully-laid-out-label roster as a gallery in each grammar (reproducible per grammar); confirm the legibility linter's report for the roster is identical regardless of which grammar displays it and unchanged in its pre-attentive governance versus the same roster with 198-era styled labels; confirm the skill documents the layout/decoration capabilities and passes the skill-parity check.

**Acceptance Scenarios**:

1. **Given** a fully-laid-out-label roster and a selected grammar, **When** a review gallery is produced, **Then** every unit's label is drawn with its alignment/decoration in that grammar and the board is byte-reproducible (under a fixed measurement provider).
2. **Given** the same roster scored with 198-era styled labels and with aligned/decorated labels, **When** the legibility linter scores each, **Then** its verdict is grammar-independent and the addition of layout/decoration does not alter the governance of the existing pre-attentive channels.
3. **Given** the design-loop skill's documentation, **When** an agent reads it, **Then** full rich-text layout is described as an opt-in inspection-detail identity channel that requires the real measurer for tofu-free output, with guidance to keep paragraphs short, use a restrained alignment/decoration set, avoid layouts/decorations that impersonate faction/state encodings or crowd the region, and that it complements, never replaces, the vector sigil.

### Edge Cases

- **None of the new capabilities used (the default)**: a `Token` whose label sets no new typographic attribute and uses default alignment with no explicit breaks MUST render **byte-identically** to spec 198; a plain label MUST stay byte-identical to spec 197; a no-label symbol MUST stay byte-identical to the pre-feature symbol (layered zero behavioural drift when the new capabilities are unused).
- **Default-valued attributes**: any new run attribute (slant / underline / strike / tracking) MAY be omitted and inherits the default (no decoration, upright, default tracking); a run that omits every new attribute is indistinguishable from (byte-identical to) the equivalent spec-198 run.
- **Default alignment**: the default alignment MUST reproduce spec 198's existing flow exactly; only a non-default alignment changes the bytes.
- **Justification of a single / last line**: the last line of a paragraph (and any line that is the only line) MUST NOT be justified (no stretched final line); justification distributes space only across lines that wrap.
- **Justification with one token / no inter-word space**: a line with no distributable inter-word gaps (a single long token) MUST fall back to the paragraph's base alignment for that line rather than clip, stretch a glyph, or overflow.
- **Empty / whitespace / empty-run label**: a label that is empty, all-whitespace, all-empty-runs, or otherwise yields no drawable glyphs is treated as no label — no text node, no exception — regardless of alignment or decoration set; interior empty/whitespace runs are normalised deterministically.
- **Decoration on overlong / wrapped runs**: underline / strike-through MUST follow each drawn run's fitted geometry per line (a run that wraps gets decoration on each of its drawn line fragments) and MUST NOT extend past the region or a clipped glyph; letter-spacing is included in per-run measurement so tracking never pushes the block past the region.
- **Too much laid-out content**: the drawn line count is capped to what the region holds; surplus is indicated (e.g. trailing ellipsis on the last drawn line) rather than overflowing vertically into the sigil/health/other channels, for every alignment.
- **Mixed sizes/weights/decorations on a line**: each line's height follows its tallest run and runs share a common baseline; decoration and tracking never cause vertical overlap or clipping, under any alignment.
- **Degenerate token (`R <= 0`) with a laid-out label**: the existing visible-placeholder rule applies and takes precedence; never a blank scene, never an exception.
- **No real measurer installed (pure-fallback path)**: the symbol's scene is still produced deterministically and the label still emits its per-run styled text nodes (with their decoration / slant / tracking and the chosen alignment recorded); tofu-free *rendering* and *wrapping / justification at measured boundaries* are properties of the render edge. The feature MUST NOT make the pure library depend on a measurer being installed, and MUST NOT throw when none is.
- **Determinism vs. measurement provider**: for a fixed measurement provider, identical laid-out `Token` ⇒ byte-identical scene. Where per-run wrapping, justification, and block geometry are computed from measured text, the result is a deterministic function of the resolved measurement; the contract is reproducibility under a fixed provider, not identical bytes across different providers.
- **Layout vs. pre-attentive channels**: alignment and decoration are author-chosen, but a layout/decoration crafted to mimic the faction or state channel encoding can mislead. The label remains an inspection-detail channel; avoiding such collisions is a design-loop guidance caveat (skill doc), not a silently enforced runtime rule, and it does NOT change the linter's pre-attentive governance.
- **Long Unicode / non-Latin runs**: each run is treated as an opaque short string in its own style; wrapping, justification, decoration, fitting, and tofu-free behaviour follow whatever the installed real measurer/shaper supports (no new shaping/bidi responsibility is taken on by this feature).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The existing styled-run label channel (`LabelText.Rich` of styled runs) MUST support **per-paragraph alignment** — at least **leading, centre, trailing, and justify** — applied to the drawn lines **within the existing per-grammar label region**, carried through the **same shared label channel** so a single `'stats -> Token` mapping drives every grammar unchanged; no new per-grammar mapping and no second label channel.
- **FR-002**: A styled (`Rich`) label MUST support **explicit paragraph and line structure** (hard breaks that the author controls, beyond the measured soft-wrap of 197/198), so multi-line styled emphasis is first-class; each paragraph MAY carry its own alignment.
- **FR-003**: Each run MUST support the **typographic attributes beyond colour/weight/size** that complete "full rich text": at least **slant (italic), underline, strike-through, and letter-spacing (tracking)** — in addition to the existing colour / weight / size. Any new attribute MAY be omitted and inherits its default (no decoration, upright, default tracking).
- **FR-004**: All new capabilities MUST be **opt-in and layered-additive**: a `Token` whose label sets no new typographic attribute and uses default alignment with no explicit breaks MUST render **byte-identically** to spec 198; a plain label MUST stay byte-identical to spec 197; a `Token` with **no** label MUST stay byte-identical to the pre-feature symbol (layered zero behavioural drift when the new capabilities are unused). Default alignment MUST reproduce spec 198's existing flow exactly.
- **FR-005**: Each grammar (Token, Badge, Ring) MUST draw the laid-out / decorated label in the **same per-grammar label region** already used, in reading order, such that — when a new capability is present — it observably alters the output and does **not overlap** the sigil, health, or other channels.
- **FR-006**: When the symbol is rendered through the headless render bridge (which installs the real text measurer/shaper), **every run** of the label MUST be drawn with **real glyphs — tofu-free** — in its own style including the new decoration / slant / tracking, and legible at the target on-board size. *Legible* is operationalized as **tofu-free** (`Missing = false` for every covered glyph of every drawn run) at the run's drawn size, with the design-loop eyeball pass as the human legibility check; the automated assertion is the tofu-free property, not a measured contrast/size threshold.
- **FR-007**: The laid-out label MUST be **fitted** to its region using real text measurement applied **per run in that run's own style** (letter-spacing included in measurement): lines MUST wrap at measured boundaries, **justification MUST distribute space using measured widths** with the last line of each paragraph left un-justified, each line's height MUST follow its tallest run with runs sharing a common baseline, the number of drawn lines MUST be **capped** to what the region holds, surplus content MUST degrade gracefully (e.g. truncation with an ellipsis on the last drawn line), and the drawn block MUST NOT clip mid-glyph and MUST NOT overflow the footprint horizontally or vertically into adjacent channels — under every alignment.
- **FR-008**: **Underline and strike-through** MUST follow each drawn run's fitted geometry per line (a run that wraps is decorated on each drawn line fragment), MUST share the line baseline/metrics, and MUST NOT extend past the region or a clipped glyph. **Justification** MUST NOT stretch glyphs or clip; a line with no distributable inter-word space MUST fall back to the paragraph's base alignment.
- **FR-009**: A label that is **empty, all-whitespace, all-empty-runs, or otherwise reduces to no drawable glyphs** MUST be treated as no label (no text node, equivalent output to having no label) and MUST NOT raise, regardless of the alignment or decoration set; interior empty/whitespace runs MUST be normalised deterministically rather than drawn as wasted gaps.
- **FR-010**: A **degenerate token** (`R <= 0`) carrying a laid-out / decorated label MUST still degrade to the existing **visible placeholder** and MUST NOT raise; the placeholder rule takes precedence over the label.
- **FR-011**: The laid-out label MUST be **pure and deterministic** with respect to a fixed text-measurement provider: identical laid-out `Token` ⇒ identical scene ⇒ identical canonical bytes, in-process and across processes, with no wall-clock, randomness, or IO performed by the library. Per-run wrapping, justification, alignment, and block layout MUST be a deterministic function of the resolved measurement.
- **FR-012**: The pure symbology library MUST **NOT depend** on a real measurer being installed: it MUST emit a deterministic scene (including each run's styled text node and the recorded alignment/decoration) on the pure-fallback path and MUST NOT throw when no measurer is present. Tofu-free *rendering* and measured wrapping/justification are properties of the render edge, not preconditions of the pure library.
- **FR-013**: A designer MUST be able to assemble a **review board** (gallery, and filmstrip where motion applies) of a laid-out-label roster in a selected grammar, reproducibly under a fixed measurement provider, with **no signature change** to the existing board/motion entry points (`render`, `galleryIn`, `filmstripIn`, `animateIn`).
- **FR-014**: The existing **legibility linter** (spec 194) MUST continue to return a **grammar-independent** verdict; if it reports the label at all, the label — however laid out or decorated — MUST be treated as an **inspection-detail** channel that does not change the governance of the existing pre-attentive channels.
- **FR-015**: Alignment, decoration, and run colours MUST be **author-supplied** from the existing scene/label vocabulary; the feature MUST NOT silently re-map or reject them at runtime. The guidance that layouts/decorations should not impersonate the faction/state pre-attentive encodings or crowd the region is a **design-loop / skill caveat**, not an enforced runtime rule, and MUST NOT change the linter's pre-attentive governance (the **state/faction palettes** remain the pre-attentive channels; the label is inspection-detail).
- **FR-016**: The existing **Token, Badge, and Ring** grammars and their motion/gallery/filmstrip behaviour MUST be **unchanged** for any `Token` without a label, any `Token` whose label is plain text, **and** any `Token` whose styled label uses none of the new capabilities (zero behavioural drift); the existing channel set is otherwise untouched.
- **FR-017**: This is an additive change to the symbology package. If it alters the package's curated public surface (e.g. a new alignment type, new `LabelRun` attribute fields, or paragraph/line structure in `LabelText`), that surface baseline MUST be regenerated to capture the change, with **zero surface drift** on every other package baseline; if a capability reuses existing public surface without adding new public surface, the baselines MUST remain unchanged and that MUST be recorded. Per Constitution II, any new public type or field MUST be declared in the module's `.fsi`.
- **FR-018**: The new capabilities MUST stay in the **pure scene-only layer** for scene construction — they add no rendering, raster, GL, or IO dependency to the pure library; they depend only on the existing scene text/measurement vocabulary already available to the grammars (`measureTextResolved` / `glyphRunProof` / `FontSpec` / `Color`), and bundle **no new font files**.
- **FR-019**: The following remain **out of scope**: inline images, hyperlinks, bullet / numbered **lists**, **per-glyph** styling, automatic generation or styling of labels from stats without a human in the loop, label-bound **motion** / animating runs, advanced bidirectional or complex-script typography beyond what the installed measurer already supports, any new **GPU/compute** path, and shipping **new font files**. The feature uses whatever font/measurer the render edge already provides; underline/strike/italic are achieved with the measurer's existing metrics/glyphs, not new font assets.
- **FR-020**: The `fs-gg-symbology` design-loop skill MUST document full rich-text layout — when to use alignment / justification / explicit breaks / decoration, the supported run attributes (italic / underline / strike-through / letter-spacing on top of colour / weight / size), that they require the real measurer for tofu-free output, that authors should keep paragraphs short and use a restrained alignment/decoration set, that layouts/decorations must not impersonate faction/state encodings or crowd the region, how surplus content degrades, and that they complement rather than replace the vector sigil — authored canonically and mirrored to every skill tree, passing the skill-parity check.

### Key Entities *(include if feature involves data)*

- **Token (existing, extended)**: the fixed channel-set value describing one unit's encoded state. Its existing optional label channel is **enriched** so a styled label may carry alignment, explicit paragraph/line structure, and per-run decoration / slant / tracking; all other channels and their meanings are unchanged. A `Token` with no label, a plain label, or a styled label using none of the new capabilities behaves exactly as before this feature.
- **Styled label run (existing, extended)**: one span of label text plus its style. Today it carries colour / weight / size; this feature adds **slant (italic), underline, strike-through, and letter-spacing (tracking)**, each optional and inheriting a no-decoration / upright / default-tracking default when omitted. Inspection-detail, complementary to the vector sigil; sited per grammar so runs never collide with other channels; tofu-free when rendered through the real measurer.
- **Label layout (new channel detail)**: the per-paragraph **alignment** (leading / centre / trailing / justify) and the **explicit paragraph / line structure** applied to a styled label within its per-grammar region. Default alignment with no explicit breaks reproduces the spec-198 flow exactly. Inspection-detail; operates strictly inside the existing region and fit/cap rules.
- **Grammar (existing)**: the selectable form factor (Token / Badge / Ring). Each sites the label (now possibly aligned and decorated) in its own region; the label is part of the one shared vocabulary, so the choice of grammar changes only where/how the label is drawn, never the `ChannelMap`.
- **Text measurement provider (existing, at the render edge)**: the real measurer/shaper installed by the rendering edge that makes label glyphs tofu-free and drives **per-run** measurement, wrapping, **justification**, decoration geometry, and block fit. The pure library does not require it; the render bridge supplies it.
- **Legibility report (existing, spec 194)**: the governance artifact; grammar-independent, and unchanged in its pre-attentive-channel governance by the addition of the inspection-only label, however laid out or decorated.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A roster's `'stats -> Token` mapping can attach a label that sets **per-paragraph alignment** (leading / centre / trailing / justify), **explicit paragraph/line structure**, and per-run **italic / underline / strike-through / letter-spacing** (on top of colour / weight / size) that renders in **all three** grammars with **zero** additional per-grammar mapping and **zero** signature change to the existing board/motion entry points.
- **SC-002**: A laid-out / decorated symbol rendered through the render bridge shows **every run** with **real glyphs (tofu-free)** in its own style including the new decoration / slant / tracking; two labels with the **same characters** but **different alignment or decoration** produce **observably distinct** rendered output. *Distinguishable* and *legible* are operationalized as **tofu-free glyph runs** (`Missing = false`) on every drawn run plus **observably distinct rendered output** for distinct layout/decoration — not a measured contrast/size metric — with the eyeball-loop as the human check.
- **SC-003**: A `Token` whose label uses **none** of the new capabilities renders **byte-identically** to spec 198; a plain label renders byte-identically to spec 197; a no-label symbol renders byte-identically to the pre-feature symbol — in every grammar (layered zero behavioural drift). Default alignment reproduces the spec-198 flow byte-for-byte.
- **SC-004**: For a fixed measurement provider, rendering the same laid-out / decorated `Token` twice — same process and separate process — yields **byte-identical** scene data (100% reproducible), including justified lines.
- **SC-005**: An overlong / over-numerous / justified / decorated label stays within the symbol footprint with **zero** mid-glyph clips, **zero** horizontal or vertical overflow into adjacent channels, mixed-size lines sized to their tallest run, decoration confined to its drawn run geometry, a **bounded** drawn line count, and the last line of each paragraph left un-justified; an empty/whitespace/empty-run label and a degenerate-token-with-label both produce **zero** exceptions.
- **SC-006**: The legibility linter's verdict for a fixed roster is **identical** regardless of grammar, and is unchanged in its governance of the existing pre-attentive channels by the presence of laid-out / decorated labels.
- **SC-007**: Surface baselines reflect the change exactly — only the symbology package's baseline moves if new public surface is added (else all baselines are unchanged and that is recorded), with **zero drift** on all other package baselines; the design-loop skill documents the full rich-text layout capabilities and passes the skill-parity check with **zero** critical/high findings.

## Assumptions

- **How layout and decoration are expressed**: the label stays the **same shared label channel** specs 196–198 introduced. The exact field shapes — how alignment and explicit paragraph/line structure are added (e.g. an alignment value on the `Rich` label and/or per-paragraph grouping, a line-break run marker, or a paragraph list), and which `LabelRun` fields carry the new slant / underline / strike / tracking attributes — are **implementation / design-loop details resolved at planning**, not contracts. The binding contracts are FR-004 (layered opt-in, zero drift when unused; default alignment = 198 flow), FR-001/FR-002/FR-003 (the capability set: alignment incl. justify, explicit breaks, italic/underline/strike/tracking), FR-005 (sited, observable, non-overlapping), FR-007/FR-008 (per-run fitted, justified by measured widths, capped, no clip/overflow, decoration follows run geometry), FR-011 (deterministic), and FR-015 (author-supplied, guidance-only governance).
- **Scope of "full rich text layout"**: interpreted as completing the two items 198 explicitly deferred (198 FR-018) — **(a)** the layout half: per-paragraph alignment incl. justification and explicit paragraph/line structure within styled runs; and **(b)** the typography half: the run attributes beyond colour/weight/size, namely italic / underline / strike-through / letter-spacing. The remaining 198-deferred items that are document-processor features ill-suited to a small unit-symbology label region — inline images, hyperlinks, lists, per-glyph styling — stay **out of scope** (FR-019), keeping the channel a bounded completion rather than a general document engine.
- **Alignment set**: at least leading, centre, trailing, and justify, applied per paragraph within the existing region. Vertical alignment of the block within the region is assumed to remain the existing 197/198 siting (no new vertical-alignment control) unless trivially free; horizontal alignment / justification is the deliverable.
- **Decoration is metric-based, not new fonts**: italic / underline / strike-through / letter-spacing are produced from the installed measurer's existing glyph metrics and synthetic slant/decoration where the renderer already supports them — **no new font files** (FR-019). Tofu-free decoration is a render-edge property like all prior label work.
- **Tofu-free, measured wrapping, and justification are render-edge properties**: the pure library emits a deterministic scene with per-run styled text nodes and the recorded alignment/decoration; **legible, tofu-free glyphs**, **wrapping at measured boundaries**, and **measured justification** come from the real measurer/shaper that the render bridge already installs. The pure library neither installs nor requires a measurer and never throws without one.
- **Determinism is provider-relative**: identical laid-out `Token` ⇒ byte-identical scene **under a fixed measurement provider**. Where wrapping, justification, and block geometry are computed from measured text, the output is a deterministic function of the resolved measurement; the feature does not promise identical bytes across *different* providers, only reproducibility under a fixed one.
- **Colour / governance policy**: alignment, decoration, and run colours are author-chosen from the existing vocabulary; the symbology colour policy's pre-attentive channels remain the **faction/state palettes**. The label stays inspection-detail and the linter's governance is unchanged; "don't impersonate the pre-attentive encodings or crowd the region" is a documented design-loop caveat, not an enforced runtime constraint (FR-015).
- **Reused, unchanged**: the three grammars, the label channel and its per-grammar regions / wrap-cap-placeholder behaviour (196/197/198), the styled-run model (198), the legibility linter (194), the headless render bridge (which installs the real measurer), the scene text/measurement vocabulary (`measureTextResolved`/`glyphRunProof`/`FontSpec`/`Color`), the board/motion entry points, and the design-loop skill scaffolding all exist and are reused as-is; this feature adds only the label's alignment / explicit-break layout, the per-run decoration / slant / tracking attributes, their per-run measurement / justification / siting, tests, any regenerated surface baseline, and the skill's full-rich-text documentation.
- **One vocabulary, one enriched channel**: no new grammars, no new label channel, no auto-generation, and no change to the channel-capacity governance beyond noting the label as inspection-detail — layout and decoration as an enrichment of a single opt-in channel, not a new subsystem.
