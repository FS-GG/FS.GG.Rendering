# Feature Specification: Symbology Auto-Label & Label-Bound Motion (channel-projected labels + motion-timeline label animation)

**Feature Branch**: `200-auto-label-bound-motion`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "auto label and label bound motion" — the next two **deferred symbology backlog items**. The label channel was built up across specs 196–199: spec 196 ([label / glyph-text channel](../196-symbology-label-text/spec.md)) shipped a single-line identity label, spec 197 ([multi-line / paragraph label](../197-symbology-multiline-label/spec.md)) widened it to several lines, spec 198 ([rich-text label runs](../198-symbology-rich-text-label/spec.md)) let the label carry styled runs (per-run colour / weight / size), and spec 199 ([full rich-text layout](../199-rich-text-layout/spec.md)) completed the typography with per-paragraph alignment / justification, explicit paragraph-line structure, and per-run italic / underline / strike-through / letter-spacing. Spec 199 then explicitly deferred (199 FR-019) two items this feature delivers: **automatic generation of labels from a unit's encoded state without hand-authoring every label** ("auto label") and **label-bound motion** (animating the label's runs over the symbology motion timeline). This feature lifts both — still tofu-free, still fitted to the region, still an opt-in inspection-detail channel, still **byte-identical** when neither capability is used, and still keeping the **per-game stat mapping out of the library**.

## Background

The symbology library encodes one unit's state into a fixed **channel grammar** (`Token -> Scene`) rendered in three grammars (Token, Badge, Ring). A deliberate architectural rule has held throughout: the **per-game `'stats -> Token` mapping lives outside the library** — the library renders *channels*, never a specific game's raw stats. The label channel (196–199) is an **opt-in, inspection-detail** identity channel layered on top of the pre-attentive faction / state / health channels: it is author-supplied, fitted to a per-grammar region with real per-run text measurement, tofu-free at the render edge, deterministic under a fixed measurement provider, and **byte-identical to the pre-feature symbol when absent**. The symbology board also already has a **motion timeline** — the `filmstripIn` / `animateIn` entry points alongside `render` / `galleryIn` — that samples a roster across deterministic motion phases for the design-loop's render→eyeball→tweak review.

Two limits remain after 199, and they are exactly what 199 FR-019 deferred:

1. **No auto-derived label.** Every label is **hand-authored** today. To label a roster, a designer must attach explicit text / styled runs to each `Token` in their per-game mapping. There is no library-provided, opt-in way to **project a label from the `Token`'s already-encoded channels** (e.g. a compact textual readout of the unit's state / tier / faction code) so common labels need not be written by hand for every unit — while still keeping the per-game stat semantics out of the library and the designer in the loop.

2. **No label-bound motion.** The label is **static**. The symbology motion timeline (`filmstripIn` / `animateIn`) animates the symbol, but the label's runs do not participate — a label cannot type-on, fade-in, pulse, or scroll an overflowing line as the motion phase advances. Motion that draws the eye to a changing identity (an alerting callsign, a freshly-changed status) is unavailable without faking it frame by frame.

This feature lifts both limits as a **bounded completion** of the existing label channel, not a new subsystem:

- **Auto label** adds an **opt-in projection** that derives a styled label (using the 196–199 rich-text vocabulary) **from the `Token`'s own channels** — never from a game's raw stats, which stay out of the library. It is deterministic, previewable in the design loop, and **overridable**: an author-supplied explicit label always wins, so the "human in the loop" that 199 FR-019 called for is preserved (opt-in to request it, override to replace it, eyeball-loop to review it).
- **Label-bound motion** lets the label's runs **animate over the existing motion timeline** — sampled as a pure function of the motion phase, so a filmstrip / animation of the roster draws each label's motion deterministically, tofu-free, fitted to the region, and byte-identical to a static label at the identity phase.

Everything stays **opt-in and additive**: a `Token` that requests neither auto-label nor label-bound motion renders **byte-identically** to spec 199 (and through 199's own zero-drift chain, byte-identical down to a plain label, and a no-label symbol byte-identical to the pre-feature symbol). Both capabilities stay in the **pure scene-only layer**: deterministic, no wall-clock / randomness / IO in the library, no new GPU/compute path, no new font files, and no dependency on a measurer being installed (tofu-free rendering and measured fit remain render-edge properties).

What stays **out of scope** (still deferred, FR-021): per-game stat → label semantics inside the library (the mapping stays the caller's), inline images, hyperlinks, bullet / numbered **lists**, **per-glyph** styling, advanced bidirectional / complex-script typography beyond what the installed measurer already supports, any new **GPU/compute** path, and shipping **new font files**.

## User Scenarios & Testing *(mandatory)*

A designer has a large roster whose units share a common labelling convention — a faction code and a tier readout — and a few whose changing status should *draw the eye* on the live board. With specs 196–199 the designer must hand-author a styled label for **every** unit and can only show it **statically**. This feature lets the same `'stats -> Token` mapping (a) **opt a `Token` into an auto-derived label** projected from its own encoded channels (still overridable by an explicit label), and (b) **bind a label to the motion timeline** so its runs animate (type-on / fade / pulse / overflow-scroll) deterministically across a filmstrip or animation — all drawn with real glyphs (never tofu), fitted so the label never overflows or collides with other channels, in every grammar. A `Token` that uses **neither** capability renders **byte-identically** to spec 199; a plain label is byte-identical to 197; a no-label symbol is byte-identical to the pre-feature symbol — the additions are purely additive and opt-in, and the per-game stat semantics stay outside the library.

### User Story 1 - Auto-derived labels projected from a Token's channels (Priority: P1)

A designer opts a `Token` into an **auto-derived label** instead of hand-authoring one. The library projects a styled label **from that `Token`'s own encoded channels** (e.g. a compact code / tier / state readout) using the existing rich-text label vocabulary, deterministically and tofu-free, in every grammar. The projection never reaches into a game's raw stats — only the `Token`'s channels — so the per-game mapping stays outside the library. The result is **overridable**: if the author also supplies an explicit label, the explicit label wins, so auto-label only fills where the designer asked for it and did not write one. A `Token` that does **not** opt in renders **byte-identically** to spec 199.

**Why this priority**: It is the core "auto label" payload and the smallest independently demonstrable slice — a pure `Token -> label` projection in the scene layer that needs no motion timeline. Shipping it alone removes the per-unit hand-authoring burden for common labels while preserving the library's "no per-game stats" rule and the human-in-the-loop override.

**Independent Test**: Build `Token`s that (a) opt into auto-label with no explicit label, (b) opt into auto-label **and** supply an explicit label, and (c) do neither; render each in each grammar through the headless render bridge, and confirm (1) the auto-label `Token` (a) renders a styled label projected from its channels, drawn with real glyphs (non-tofu), fitted to the region; (2) the override `Token` (b) renders the **explicit** label, not the projection (explicit wins); (3) the opt-out `Token` (c) is **byte-identical** to spec 199; and (4) two `Token`s whose channels differ in a projected field produce **observably different** auto-labels, while two `Token`s with identical channels produce **byte-identical** auto-labels (the projection is a deterministic pure function of the channels).

**Acceptance Scenarios**:

1. **Given** a `Token` opted into an auto-derived label with no explicit label, **When** it is rendered in any grammar through the render bridge, **Then** a styled label projected from that `Token`'s channels is drawn in reading order with real glyphs (tofu-free), fitted to the per-grammar region, and legible at the target on-board size.
2. **Given** a `Token` opted into an auto-derived label that **also** carries an explicit author-supplied label, **When** it is rendered, **Then** the **explicit** label is drawn (explicit overrides auto) and no exception is raised.
3. **Given** two `Token`s identical except for one channel value that the projection reads, **When** both are rendered, **Then** their auto-labels differ in a way attributable to that channel; **and given** two `Token`s with identical channels, their auto-labels are byte-identical (deterministic projection).
4. **Given** a `Token` that does **not** opt into an auto-derived label (and carries no explicit label), **When** it is rendered, **Then** its scene is byte-identical to spec 199 (and through 199's chain, to the pre-feature no-label symbol).
5. **Given** the same auto-label `Token` rendered twice (in-process and across processes) under the same text-measurement provider, **When** the two scenes are compared, **Then** they are byte-identical (determinism).

### User Story 2 - Label-bound motion over the symbology motion timeline (Priority: P2)

A designer binds a label to the **existing motion timeline** so its runs **animate** as the motion phase advances — for example a callsign that **types on**, a status that **fades / pulses** to alert, or an overflowing line that **scrolls** (ticker) within the region — rather than sitting static. The animation is sampled as a **pure function of the motion phase** the board already supplies (`filmstripIn` / `animateIn`), so a filmstrip or animation of the roster draws each label's motion **deterministically**, tofu-free, and **fitted** (the animated label never overflows the region or collides with the sigil / health / other channels at any phase). At the **identity phase** (the rest frame), a motion-bound label is drawn **byte-identically** to the equivalent static spec-199 label, so motion is purely additive and a label that binds **no** motion is unchanged.

**Why this priority**: Motion is the "label-bound motion" half and the half that needs the fit machinery to stay trustworthy across **every** phase — a typed-on or scrolling line that overflows mid-animation is worse than a static label. It builds on the label rendering proven by US1 (and 196–199) and on the existing motion timeline, so it follows.

**Independent Test**: Render `filmstripIn` / `animateIn` sequences of `Token`s whose labels bind each motion kind (type-on / fade / pulse / overflow-scroll), including labels longer than the region holds; confirm (1) each label's drawn state is a deterministic function of the motion phase — the same phase yields byte-identical label output in-process and across processes; (2) at the identity / rest phase the motion-bound label is byte-identical to the static spec-199 label; (3) the animated label stays **fitted** at every sampled phase — no mid-glyph clip, no overflow into adjacent channels, capped line count preserved; and (4) a `Token` whose label binds **no** motion is byte-identical to spec 199 across the whole sequence.

**Acceptance Scenarios**:

1. **Given** a `Token` whose label binds a motion (e.g. type-on or fade), **When** the roster is rendered as a `filmstripIn` / `animateIn` sequence, **Then** the label's drawn state advances with the motion phase, each frame tofu-free and fitted to the region, and the symbol's other channels are unaffected.
2. **Given** a motion-bound label sampled at the **identity / rest phase**, **When** that frame is rendered, **Then** the label is byte-identical to the equivalent static spec-199 label (motion is additive; rest = static).
3. **Given** a label that binds an overflow-scroll (ticker) motion over content longer than the region, **When** the sequence is rendered, **Then** the line scrolls within the region across phases without ever clipping mid-glyph or overflowing into adjacent channels, and the drawn line count stays capped.
4. **Given** the same motion-bound `Token` and motion phase rendered twice (in-process and across processes) under the same measurement provider, **When** the two frames are compared, **Then** they are byte-identical (deterministic per-phase sampling; no wall-clock / randomness).
5. **Given** a `Token` whose label binds **no** motion, **When** the roster is rendered across the motion timeline, **Then** every frame's label is byte-identical to spec 199 (zero drift when motion unused).

### User Story 3 - Auto-labelled / motion-bound rosters on review boards, governed unchanged (Priority: P3)

A designer assembles a review board (gallery / motion filmstrip) of a roster that mixes auto-derived and motion-bound labels in any grammar to A/B legibility, and runs the existing legibility linter (spec 194). Auto-labelled and motion-bound labels render reproducibly on the board; the linter continues to return a **grammar-independent** verdict and treats the label — however derived or animated — as an **inspection-detail** channel (it may report label usage but does not change its governance of the pre-attentive channels). The design-loop skill documents auto-label and label-bound motion with their legibility caveats: keep auto-projections compact, use restrained motion, don't let an auto-label or its motion impersonate the faction / state pre-attentive encodings or crowd the region, inspection-only, requires the real measurer to be tofu-free, complements the vector sigil.

**Why this priority**: It turns auto-labelled / motion-bound labels into something the loop can compare and govern, and confirms the linter and skill keep working. Strictly additive on top of US1 / US2.

**Independent Test**: Render a roster mixing auto-derived and motion-bound labels as a gallery / filmstrip in each grammar (reproducible per grammar under a fixed measurement provider and fixed phase sampling); confirm the legibility linter's report for the roster is identical regardless of which grammar displays it and unchanged in its pre-attentive governance versus the same roster with 199-era static hand-authored labels; confirm the skill documents the auto-label / motion capabilities and passes the skill-parity check.

**Acceptance Scenarios**:

1. **Given** a roster mixing auto-derived and motion-bound labels and a selected grammar, **When** a review gallery / filmstrip is produced, **Then** every unit's label is drawn (projected and/or animated) in that grammar and the board is byte-reproducible under a fixed measurement provider and fixed phase sampling.
2. **Given** the same roster scored with 199-era static hand-authored labels and with auto-derived / motion-bound labels, **When** the legibility linter scores each, **Then** its verdict is grammar-independent and the addition of auto-derivation / motion does not alter the governance of the existing pre-attentive channels.
3. **Given** the design-loop skill's documentation, **When** an agent reads it, **Then** auto-label and label-bound motion are described as opt-in inspection-detail identity capabilities that require the real measurer for tofu-free output, that keep per-game stat semantics out of the library, with guidance to keep auto-projections compact and motion restrained, avoid auto-labels / motion that impersonate faction/state encodings or crowd the region, and that they complement, never replace, the vector sigil.

### Edge Cases

- **Neither capability used (the default)**: a `Token` that opts into neither an auto-derived label nor label-bound motion MUST render **byte-identically** to spec 199; a plain hand-authored label MUST stay byte-identical to spec 197; a no-label symbol MUST stay byte-identical to the pre-feature symbol (layered zero behavioural drift when the new capabilities are unused).
- **Explicit label overrides auto-label**: when a `Token` both opts into an auto-derived label and carries an explicit author-supplied label, the **explicit** label MUST be drawn and the projection ignored; there MUST be exactly one resolved label, never two stacked.
- **Auto-label projects to nothing drawable**: if the projection of a `Token`'s channels yields empty / all-whitespace / no-drawable-glyph text (e.g. a channel set the projection reads is degenerate), it MUST be treated as **no label** (no text node, no exception), exactly as an empty hand-authored label is (199/197 rule).
- **Auto-label fitting**: a projected label is fitted to the per-grammar region by the **same** real per-run measurement, wrap, cap, truncate, and placeholder rules as a hand-authored label (196–199) — it MUST NOT clip mid-glyph or overflow into adjacent channels, regardless of how long the projected text is.
- **Determinism of the projection**: the auto-label is a **pure deterministic function of the `Token`'s channels** only — no wall-clock, randomness, IO, or hidden global state; identical channels ⇒ identical projected label ⇒ identical bytes (under a fixed measurement provider).
- **No per-game stats in the projection**: the projection reads only the `Token`'s **encoded channels**, never a game's raw stats; the per-game `'stats -> Token` mapping stays the caller's responsibility and is not imported into the library by this feature.
- **Motion at the identity / rest phase**: a motion-bound label sampled at the rest phase MUST be byte-identical to the equivalent static spec-199 label; only a non-rest phase changes the bytes, so binding a motion does not perturb the static frame.
- **Motion stays fitted at every phase**: an animated label (type-on partial reveal, scroll offset, pulse scale/alpha) MUST remain fitted to the region at **every** sampled phase — no mid-glyph clip, no overflow into the sigil / health / other channels, capped line count preserved; an overflow-scroll MUST scroll within the region, never beyond it.
- **Deterministic per-phase sampling**: the animated label state MUST be a **deterministic function of the motion phase** the board supplies; the same phase yields byte-identical output in-process and across processes — no wall-clock or randomness in the library (consistent with the existing `filmstripIn` / `animateIn` determinism).
- **Motion bound to an empty / absent label**: binding motion to a label that resolves to no drawable glyphs is a no-op (no text node, no exception) at every phase; motion never fabricates a label.
- **Auto-label + motion together**: a `Token` MAY both auto-derive its label and bind that label to motion; the projection resolves first (pure function of channels), then the resolved label is animated per phase — composition MUST stay deterministic, fitted, and tofu-free.
- **Degenerate token (`R <= 0`) with an auto / motion label**: the existing **visible-placeholder** rule applies and takes precedence; the auto-label / motion is suppressed, never a blank scene, never an exception.
- **No real measurer installed (pure-fallback path)**: the symbol's scene is still produced deterministically; the (resolved) label still emits its styled text nodes (projected and/or with its motion phase recorded). Tofu-free *rendering* and *measured fit / wrapping* are render-edge properties. The feature MUST NOT make the pure library depend on a measurer being installed, and MUST NOT throw when none is.
- **Linter governance unchanged**: an auto-derived or motion-bound label, however projected or animated, remains an **inspection-detail** channel; it MUST NOT change the legibility linter's governance of the pre-attentive (faction / state / health) channels, and the verdict MUST stay grammar-independent.
- **Impersonation is a guidance caveat**: an auto-projection or motion crafted to mimic the faction / state channel encoding can mislead; avoiding that is a **design-loop / skill caveat**, not a silently enforced runtime rule, and does NOT change the linter's pre-attentive governance.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The label channel MUST support an **opt-in auto-derived label** that the library **projects from a `Token`'s own encoded channels** into a styled label using the existing rich-text label vocabulary (196–199), carried through the **same shared label channel** so a single `'stats -> Token` mapping drives every grammar unchanged; no new per-grammar mapping and no second label channel.
- **FR-002**: The auto-label projection MUST read **only the `Token`'s encoded channels**, never a game's raw stats; the per-game `'stats -> Token` mapping MUST remain the caller's responsibility and MUST NOT be imported into the pure library by this feature.
- **FR-003**: The auto-derived label MUST be **overridable**: when a `Token` both opts into an auto-derived label and carries an explicit author-supplied label, the **explicit** label MUST be drawn and the projection ignored — exactly one resolved label, explicit wins (human-in-the-loop override).
- **FR-004**: The auto-label projection MUST be a **pure, deterministic function of the `Token`'s channels** only — no wall-clock, randomness, IO, or hidden global state; identical channels ⇒ identical projected label. If the projection yields empty / all-whitespace / no-drawable-glyph text it MUST be treated as **no label** (no text node, no exception).
- **FR-005**: The label channel MUST support **opt-in label-bound motion** that animates the resolved label's runs (e.g. type-on, fade, pulse, overflow-scroll) over the **existing symbology motion timeline**, with **no signature change** to the existing board/motion entry points (`render`, `galleryIn`, `filmstripIn`, `animateIn`).
- **FR-006**: The animated label state MUST be a **deterministic function of the motion phase** the board supplies — the same phase yields identical output in-process and across processes, with no wall-clock or randomness in the library (consistent with the existing `filmstripIn` / `animateIn` determinism).
- **FR-007**: At the **identity / rest phase**, a motion-bound label MUST be drawn **byte-identically** to the equivalent static spec-199 label; only a non-rest phase changes the bytes (motion is additive; binding a motion does not perturb the static frame).
- **FR-008**: Both capabilities MUST be **opt-in and layered-additive**: a `Token` that opts into **neither** auto-label nor label-bound motion MUST render **byte-identically** to spec 199; a plain hand-authored label MUST stay byte-identical to spec 197; a `Token` with **no** label MUST stay byte-identical to the pre-feature symbol (layered zero behavioural drift when the new capabilities are unused).
- **FR-009**: Each grammar (Token, Badge, Ring) MUST draw the resolved (projected and/or animated) label in the **same per-grammar label region** already used, in reading order, such that — when a new capability is present — it observably alters the output and does **not overlap** the sigil, health, or other channels at any phase.
- **FR-010**: When the symbol is rendered through the headless render bridge (which installs the real text measurer/shaper), **every run** of the resolved label MUST be drawn with **real glyphs — tofu-free** — in its own style, at every sampled motion phase, and legible at the target on-board size. *Legible* is operationalized as **tofu-free** (`Missing = false` for every covered glyph of every drawn run) at the run's drawn size, with the design-loop eyeball pass as the human legibility check.
- **FR-011**: The resolved label (projected and/or animated) MUST be **fitted** to its region using real per-run text measurement under the **same** wrap / justify / cap / truncate / placeholder rules as a hand-authored 199 label: it MUST NOT clip mid-glyph and MUST NOT overflow the footprint horizontally or vertically into adjacent channels — at **every** sampled phase. An overflow-scroll (ticker) MUST scroll within the region, never beyond it; the drawn line count MUST stay capped.
- **FR-012**: A label that **resolves to no drawable glyphs** (auto-projection of degenerate channels, an empty hand-authored label, all-whitespace, all-empty-runs) MUST be treated as no label (no text node) and MUST NOT raise, at every motion phase; binding motion to such a label is a no-op (motion never fabricates a label).
- **FR-013**: A `Token` MAY **both** auto-derive its label and bind that label to motion; the projection MUST resolve first (pure function of channels), then the resolved label MUST be animated per phase — composition staying deterministic, fitted, and tofu-free.
- **FR-014**: A **degenerate token** (`R <= 0`) carrying an auto / motion label MUST still degrade to the existing **visible placeholder** and MUST NOT raise; the placeholder rule takes precedence over the auto-label / motion.
- **FR-015**: Both capabilities MUST be **pure and deterministic** with respect to a fixed text-measurement provider and a fixed motion phase: identical inputs ⇒ identical scene ⇒ identical canonical bytes, in-process and across processes, with no wall-clock, randomness, or IO performed by the library.
- **FR-016**: The pure symbology library MUST **NOT depend** on a real measurer being installed: it MUST emit a deterministic scene (including the resolved label's styled text nodes and the recorded motion phase) on the pure-fallback path and MUST NOT throw when no measurer is present. Tofu-free *rendering* and measured fit are properties of the render edge, not preconditions of the pure library.
- **FR-017**: A designer MUST be able to assemble a **review board** (gallery, and filmstrip / animation for motion) of an auto-labelled / motion-bound roster in a selected grammar, reproducibly under a fixed measurement provider and fixed phase sampling, with **no signature change** to the existing board/motion entry points.
- **FR-018**: The existing **legibility linter** (spec 194) MUST continue to return a **grammar-independent** verdict; the label — however derived or animated — MUST be treated as an **inspection-detail** channel that does not change the governance of the existing pre-attentive (faction / state / health) channels.
- **FR-019**: The new capabilities MUST stay in the **pure scene-only layer** for scene construction — they add no rendering, raster, GL, or IO dependency to the pure library; they depend only on the existing scene text/measurement and motion vocabulary already available to the grammars (`measureTextResolved` / `glyphRunProof` / `FontSpec` / `Color` / the existing motion-phase sampling), and bundle **no new font files**.
- **FR-020**: This is an additive change to the symbology package. If it alters the package's curated public surface (e.g. an auto-label opt-in flag / projection descriptor, a motion-binding field on the label), that surface baseline MUST be regenerated to capture the change, with **zero surface drift** on every other package baseline; if a capability reuses existing public surface without adding new public surface, the baselines MUST remain unchanged and that MUST be recorded. Per Constitution II, any new public type or field MUST be declared in the module's `.fsi`.
- **FR-021**: The following remain **out of scope**: per-game stat → label semantics inside the library (the `'stats -> Token` mapping stays the caller's), inline images, hyperlinks, bullet / numbered **lists**, **per-glyph** styling, advanced bidirectional or complex-script typography beyond what the installed measurer already supports, any new **GPU/compute** path, and shipping **new font files**. The feature uses whatever font / measurer / motion timeline the render edge already provides.
- **FR-022**: The `fs-gg-symbology` design-loop skill MUST document auto-label and label-bound motion — when to use an auto-derived label vs. a hand-authored one, that auto-labels project only from `Token` channels (never per-game stats) and are overridable, the supported motion kinds and that they sample deterministically from the motion phase, that both require the real measurer for tofu-free output, that authors should keep auto-projections compact and motion restrained, that they must not impersonate faction/state encodings or crowd the region, how surplus / overflow degrades, and that they complement rather than replace the vector sigil — authored canonically and mirrored to every skill tree, passing the skill-parity check.

### Key Entities *(include if feature involves data)*

- **Token (existing, extended)**: the fixed channel-set value describing one unit's encoded state. Its existing optional label channel gains two opt-in capabilities: a request to **auto-derive** the label from the `Token`'s own channels, and a binding that animates the resolved label over the motion timeline; all other channels and their meanings are unchanged. A `Token` that opts into neither behaves exactly as before this feature.
- **Auto-label projection (new channel detail)**: the opt-in, pure, deterministic derivation of a styled label from a `Token`'s **encoded channels** (never per-game stats), expressed in the existing 196–199 rich-text vocabulary. Overridden by an explicit author-supplied label; treated as no label when it yields no drawable glyphs. Inspection-detail; fitted by the same region / wrap / cap rules as a hand-authored label.
- **Label motion binding (new channel detail)**: the opt-in association of the resolved label with a **motion kind** (e.g. type-on / fade / pulse / overflow-scroll) sampled as a deterministic function of the existing motion phase. Byte-identical to the static label at the rest phase; fitted at every phase. Inspection-detail; reuses the existing motion timeline, no new entry points.
- **Styled label (existing, 196–199)**: the resolved label — text or styled runs with colour / weight / size / alignment / decoration / tracking — that auto-projection produces and that motion animates. Reused unchanged as the render target.
- **Grammar (existing)**: the selectable form factor (Token / Badge / Ring). Each sites the resolved (projected and/or animated) label in its own region; the label is part of the one shared vocabulary, so the choice of grammar changes only where/how the label is drawn, never the `ChannelMap`.
- **Text measurement provider (existing, at the render edge)**: the real measurer/shaper installed by the rendering edge that makes label glyphs tofu-free and drives per-run measurement, fit, and decoration geometry. The pure library does not require it; the render bridge supplies it.
- **Motion timeline / phase (existing)**: the deterministic phase sampling the board already supplies via `filmstripIn` / `animateIn`, reused as the clock the label motion binding reads from. No new clock, no wall-clock.
- **Legibility report (existing, spec 194)**: the governance artifact; grammar-independent, and unchanged in its pre-attentive-channel governance by the addition of the inspection-only auto / motion label.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A roster's `'stats -> Token` mapping can **opt a `Token` into an auto-derived label** projected from its channels and/or **bind its label to motion**, rendering in **all three** grammars with **zero** additional per-grammar mapping, **zero** signature change to the existing board/motion entry points, and **zero** per-game stat semantics added to the library.
- **SC-002**: An auto-labelled / motion-bound symbol rendered through the render bridge shows **every run** of the resolved label with **real glyphs (tofu-free)** at every sampled phase; two `Token`s whose channels differ in a projected field produce **observably distinct** auto-labels, and a motion-bound label produces **observably distinct** frames across phases — operationalized as **tofu-free glyph runs** (`Missing = false`) plus observable output differences, with the eyeball-loop as the human check.
- **SC-003**: A `Token` opting into **neither** capability renders **byte-identically** to spec 199; a plain hand-authored label renders byte-identically to spec 197; a no-label symbol renders byte-identically to the pre-feature symbol — in every grammar (layered zero behavioural drift). A motion-bound label at the rest phase reproduces the static spec-199 label byte-for-byte.
- **SC-004**: For a fixed measurement provider and fixed motion phase, rendering the same auto-labelled / motion-bound `Token` twice — same process and separate process — yields **byte-identical** scene data (100% reproducible), including each animated frame.
- **SC-005**: An auto-projected, overlong, or animated (typed-on / scrolling / pulsing) label stays within the symbol footprint with **zero** mid-glyph clips and **zero** horizontal or vertical overflow into adjacent channels at **every** sampled phase, with a **bounded** drawn line count; an explicit label always overrides the projection (exactly one resolved label); an empty/whitespace/degenerate auto-projection, a motion-bound empty label, and a degenerate-token-with-label all produce **zero** exceptions.
- **SC-006**: The legibility linter's verdict for a fixed roster is **identical** regardless of grammar, and is unchanged in its governance of the existing pre-attentive channels by the presence of auto-derived / motion-bound labels.
- **SC-007**: Surface baselines reflect the change exactly — only the symbology package's baseline moves if new public surface is added (else all baselines are unchanged and that is recorded), with **zero drift** on all other package baselines; the design-loop skill documents auto-label and label-bound motion and passes the skill-parity check with **zero** critical/high findings.

## Assumptions

- **What "auto label … from stats" means here**: the binding architectural rule is that the **per-game `'stats -> Token` mapping stays outside the library** (the symbology skill's stated boundary). The only in-library interpretation consistent with that rule is an **auto-label projected from the `Token`'s already-encoded channels** — a game-agnostic readout of the channel values the library already holds — not from a game's raw stats. So "auto label" is scoped as a channel-projection (FR-001/FR-002), and per-game label semantics stay the caller's (FR-021). The deferred "without a human in the loop" concern is met by making it **opt-in**, **overridable** by an explicit label (FR-003), and reviewed in the eyeball-loop.
- **How the capabilities are expressed**: the label stays the **same shared label channel** specs 196–199 introduced. The exact field shapes — how a `Token` opts into auto-derivation (e.g. an auto-label request / projection descriptor), what subset of channels the projection reads and in what compact form, and how a motion kind is bound to the label (e.g. a motion field on the label) — are **implementation / design-loop details resolved at planning**, not contracts. The binding contracts are FR-008 (layered opt-in, zero drift when unused), FR-002/FR-003 (channel-only projection, explicit override), FR-004 (deterministic projection), FR-005/FR-006/FR-007 (motion over the existing timeline, phase-deterministic, rest = static), FR-011 (fitted at every phase), and FR-015/FR-016 (deterministic, measurer-optional).
- **Auto-label resolution order**: an **explicit author-supplied label always wins**; the auto-projection fills only where the `Token` opted in and supplied no explicit label. There is always exactly **one resolved label** (or none), never two stacked.
- **Motion reuses the existing timeline**: label-bound motion samples the **same motion phase** the board already exposes via `filmstripIn` / `animateIn`; this feature adds **no new clock**, no wall-clock, and **no signature change** to the board/motion entry points. The default motion set is assumed to be a small, restrained vocabulary (e.g. type-on, fade, pulse, overflow-scroll); the exact set is a design-loop detail resolved at planning, with the contract being deterministic per-phase sampling and rest-phase = static.
- **Motion-bound fit holds at every phase**: the animated label is fitted by the same real per-run measurement / wrap / cap / truncate rules as a static 199 label, evaluated at each sampled phase, so a typed-on, scrolled, or pulsed label never clips or overflows mid-animation.
- **Tofu-free, measured fit, and the motion timeline are render-edge properties**: the pure library emits a deterministic scene with the resolved label's styled text nodes and the recorded phase; **legible, tofu-free glyphs**, **measured fit / wrapping**, and the **phase clock** come from the render edge that the render bridge already installs. The pure library neither installs nor requires a measurer and never throws without one.
- **Determinism is provider- and phase-relative**: identical auto/motion `Token` ⇒ byte-identical scene **under a fixed measurement provider and fixed motion phase**. The feature does not promise identical bytes across *different* providers, only reproducibility under a fixed one.
- **Colour / governance policy**: the symbology colour policy's pre-attentive channels remain the **faction / state palettes**; the auto / motion label stays inspection-detail and the linter's governance is unchanged. "Don't impersonate the pre-attentive encodings or crowd the region" is a documented design-loop caveat, not an enforced runtime constraint (FR-018).
- **Reused, unchanged**: the three grammars, the label channel and its per-grammar regions / wrap-cap-placeholder behaviour (196–199), the styled-run + full-rich-text-layout model (198/199), the legibility linter (194), the headless render bridge (which installs the real measurer), the scene text/measurement vocabulary (`measureTextResolved` / `glyphRunProof` / `FontSpec` / `Color`), the **motion timeline** (`filmstripIn` / `animateIn`) and board entry points (`render` / `galleryIn`), and the design-loop skill scaffolding all exist and are reused as-is; this feature adds only the auto-label projection, the label motion binding, their per-phase resolution / fit / siting, tests, any regenerated surface baseline, and the skill's auto-label / motion documentation.
- **One vocabulary, one enriched channel**: no new grammars, no new label channel, no new motion timeline, and no change to the channel-capacity governance beyond noting the label as inspection-detail — auto-derivation and motion as an enrichment of a single opt-in channel, not a new subsystem.
