# Feature Specification: Symbology Rich-Text Label Runs (per-run colour / weight / size)

**Feature Branch**: `198-symbology-rich-text-label`

**Created**: 2026-06-25

**Status**: Draft

**Input**: User description: "next backlog item." — the symbology **M0–M7 roadmap** in [`docs/reports/2026-06-25-12-48-agent-symbology-design-system-analysis-and-plan.md`](../../docs/reports/2026-06-25-12-48-agent-symbology-design-system-analysis-and-plan.md) is complete, so this resolves to the next **deferred symbology backlog item** from the M7 "(∞)" bucket. Spec 196 ([label / glyph-text channel](../196-symbology-label-text/spec.md)) shipped a single-line identity label and spec 197 ([multi-line / paragraph label](../197-symbology-multiline-label/spec.md)) widened it to several lines — but both draw the whole label in **one uniform style** (one ink, one weight, one fitted size) and explicitly deferred **rich-text styling (per-run colour / weight / size runs within the label)** as out of scope (197 FR-016). This feature delivers that deferred item: let a single label carry **styled runs** — e.g. a bright bold callsign followed by a dim, smaller code on the same line — still tofu-free, still fitted to the region, still opt-in **byte-identical** when the label uses no styling.

## Background

Spec 196 added an **optional identity label** (`Token.Label : string option`) sited per grammar (Token / Badge / Ring), fitted to its region with real text measurement, tofu-free at the render edge, and byte-identical to the pre-feature symbol when absent. Spec 197 widened that one channel so the label may span **several short lines** (hard breaks + measured soft-wrap), capped and fitted to the region — but every line is still drawn in a **single uniform style**: one fixed ink colour, one weight, one fitted size for the whole block.

That uniform-style constraint is the limit being lifted here. Designers running the `fs-gg-symbology` render→eyeball→tweak loop sometimes need to **emphasise part of the identity relative to the rest** — a bright, bold **callsign** with a dim, smaller **unit code** beside or beneath it; a primary name next to a secondary tag. Today the whole label is one weight/colour/size, so the two pieces compete at equal salience and the eye cannot triage them. **Rich-text runs** let the same opt-in label channel carry a short sequence of styled spans — each with its own **colour, weight, and size** — laid out and fitted with the same real-measurement discipline (now measured **per run**), so the emphasis structure is readable without harming the pre-attentive channels.

This stays a **single opt-in inspection-detail channel**, not a new subsystem. It is the natural next step after 196 (single line) → 197 (multi line) → **198 (styled runs within a line)**. No automatic generation from stats, no label-bound motion, no full rich-text layout engine (justification, inline images, hyperlinks, per-glyph styling, italic/underline/strike), no new GPU path, and no new font files — those remain deferred (FR-018).

## User Scenarios & Testing *(mandatory)*

A designer has a roster where one piece of identity should read louder than another — a callsign that should pop and a code that should sit quietly beneath it. With spec 197 the label is one uniform weight/colour/size, so both read at equal salience and the designer cannot express the hierarchy. This feature lets the same `'stats -> Token` mapping carry a label whose text is a short sequence of **styled runs** — each run a span of text with its own **colour, weight, and size** — stacked and wrapped into the symbol's label region exactly as before, each run drawn with real glyphs (never tofu) and fitted (now measuring each run in its own style) so the block never overflows the footprint, clips a glyph, or collides with another channel. A unit with **no** label still renders exactly as the pre-feature symbol; a unit whose label is **plain, unstyled text** (single- or multi-line) still renders **byte-identically** to spec 197 — rich-text runs are purely additive on top of both.

### User Story 1 - Author a label with emphasis (styled runs) (Priority: P1)

A designer attaches a label whose text is a short sequence of **styled runs** — for example a **bold, bright** callsign run followed by a **dim, smaller** code run — to a unit's encoded state and renders the symbol; the runs appear in reading order within the symbol's label region, each drawn with its **own colour, weight, and size**, each with **real glyphs (not tofu)**, in every grammar (Token, Badge, Ring). A label that is **plain unstyled text** (a single run / ordinary string, single- or multi-line) renders **byte-identically** to spec 197, and a unit with **no** label renders byte-identically to the pre-feature symbol — rich-text styling is purely additive and opt-in.

**Why this priority**: It is the core value of the thread — expressing a salience hierarchy inside the identity label so two pieces of identity can be triaged at a glance — and the smallest independently demonstrable slice. With it, a designer can show a loud callsign and a quiet code in one label. Shipping it alone closes the deferred rich-text gap.

**Independent Test**: Build `Token`s carrying (a) no label, (b) a plain unstyled label, and (c) a label of two-or-more styled runs differing in colour/weight/size; render each in each grammar through the headless render bridge, and confirm (1) the styled-run label's rasterised output shows each run drawn in its own colour/weight/size with real glyphs (non-tofu); (2) a plain unstyled label's scene is byte-identical to spec 197's label, and a no-label scene is byte-identical to the pre-feature symbol for the same channels; and (3) two `Token`s whose labels carry the **same characters** but **different run styling** produce observably different rendered output (style is a channel that changes the bytes).

**Acceptance Scenarios**:

1. **Given** a `Token` whose label carries two or more styled runs (differing colour, weight, or size), **When** it is rendered in any grammar through the render bridge, **Then** each run is drawn in reading order in its own colour/weight/size with real glyphs (tofu-free) and legible at the target on-board size.
2. **Given** two `Token`s identical except one's label is plain text and the other's label carries the same characters split into styled runs, **When** both are rendered, **Then** their scenes differ in a way attributable to the run styling (channel presence), and neither raises.
3. **Given** a `Token` whose label is plain unstyled text (no styled runs — single- or multi-line), **When** it is rendered, **Then** its scene is byte-identical to spec 197 for that token (zero drift on existing single-/multi-line label behaviour).
4. **Given** the same styled-run-labelled `Token` rendered twice (in-process and across processes) under the same text-measurement provider, **When** the two scenes are compared, **Then** they are byte-identical (determinism).

### User Story 2 - Styled runs fit the region without clipping or overflow (Priority: P2)

A designer gives a unit a label whose runs are larger, bolder, or more numerous than the symbol's label region can hold. The label is **fitted** to its region using real text measurement applied **per run in that run's own style**: runs flow and wrap at measured boundaries (a run may break across lines; the line height of each line follows the tallest run on it), the number of drawn lines is **capped** to what the region holds, and surplus content degrades gracefully (e.g. the last drawn line is truncated with an ellipsis) rather than spilling. The block never overflows the footprint horizontally or vertically, never clips a glyph mid-run, and never overlaps the sigil, health, or other channels.

**Why this priority**: A styled block that overflows into the sigil/health channels, or clips, is worse than uniform text — bold/large runs make overflow more likely and obscuring pre-attentive channels is the cardinal sin. Per-run-measured fit is what makes the styled channel trustworthy, but it builds on the basic run rendering proven by P1, so it follows.

**Independent Test**: Render a `Token` whose styled runs are far wider/taller/more numerous than its region holds, and confirm the drawn block stays within the symbol's label footprint (measured per run via the real measurer), no run is cut mid-glyph, the line count is capped, surplus is indicated by truncation rather than overflow, mixed run sizes set each line's height to its tallest run, and the block does not overlap an adjacent channel; confirm a label of all-empty/whitespace runs (or a styled label that reduces to no drawable glyphs) produces no label and no exception.

**Acceptance Scenarios**:

1. **Given** a label whose runs are wider than the region, **When** the symbol is rendered, **Then** the runs are fitted (wrapped/shrunk/truncated, each measured in its own style) to stay within the region and no run is clipped mid-glyph.
2. **Given** a label with more styled content than the region can hold, **When** the symbol is rendered, **Then** the drawn line count is capped to the region and the surplus is indicated (e.g. ellipsis on the last drawn line) without overflowing into adjacent channels.
3. **Given** a label whose runs mix sizes/weights on one line, **When** the symbol is rendered, **Then** the line's height accommodates the tallest run and runs share a common baseline without overlapping vertically.
4. **Given** a label that is empty, all-whitespace, all-empty-runs, or otherwise reduces to no drawable glyphs, **When** the symbol is rendered, **Then** no label is drawn and no exception is raised (equivalent to having no label).
5. **Given** a degenerate `Token` (`R <= 0`) that also carries a styled-run label, **When** it is rendered, **Then** the existing visible placeholder is produced and no exception is raised (the placeholder rule wins over the label).
6. **Given** a styled-run label that fits, **When** the symbol is rendered, **Then** the block sits in its designated per-grammar region and does not overlap the sigil, health, or other channels.

### User Story 3 - Styled labels on review boards, governed unchanged (Priority: P3)

A designer assembles a review board (gallery / motion filmstrip) of a styled-label roster in any grammar to A/B legibility, and runs the existing legibility linter (spec 194). Styled-run labels render reproducibly on the board; the linter continues to return a **grammar-independent** verdict and treats the label — plain or styled — as an **inspection-detail** channel (it may report label usage but does not change its governance of the pre-attentive channels). The design-loop skill documents rich-text runs with their legibility caveats: keep to a few short runs, use a restrained set of styles, **do not** colour runs so they impersonate the faction/state pre-attentive encodings, inspection-only, requires the real measurer to be tofu-free, complements the vector sigil.

**Why this priority**: It turns styled labels into something the loop can compare and govern, and confirms the linter and skill keep working. Strictly additive on top of P1/P2.

**Independent Test**: Render a styled-label roster as a gallery in each grammar (reproducible per grammar); confirm the legibility linter's report for the roster is identical regardless of which grammar displays it and unchanged in its pre-attentive governance versus the same roster with plain labels; confirm the skill documents the rich-text run channel and passes the skill-parity check.

**Acceptance Scenarios**:

1. **Given** a styled-label roster and a selected grammar, **When** a review gallery is produced, **Then** every unit's runs are drawn in that grammar and the board is byte-reproducible (under a fixed measurement provider).
2. **Given** the same roster scored with plain labels and with styled-run labels, **When** the legibility linter scores each, **Then** its verdict is grammar-independent and the addition of run styling does not alter the governance of the existing pre-attentive channels.
3. **Given** the design-loop skill's documentation, **When** an agent reads it, **Then** rich-text runs are described as an opt-in inspection-detail identity channel that requires the real measurer for tofu-free output, with guidance to keep to a few short runs, use a restrained style/colour set, avoid colours that impersonate faction/state encodings, and that it complements, never replaces, the vector sigil.

### Edge Cases

- **No label (the default)**: a `Token` with no label MUST render byte-identically to the current (pre-feature) symbol — the channel stays purely additive and opt-in (zero behavioural drift when unused).
- **Plain unstyled label**: a label expressed as ordinary text with no per-run styling (single- or multi-line) MUST render byte-identically to spec 197 — rich-text runs are additive on top of the existing uniform-style behaviour (zero drift for unstyled content).
- **Single uniform run**: a label consisting of exactly one run whose style equals the existing default label style MUST be indistinguishable from (byte-identical to) the equivalent plain label — adding the run wrapper without changing the style introduces no drift.
- **Empty / whitespace / empty-run label**: a label that is empty, all-whitespace, all-empty-runs, or otherwise yields no drawable glyphs is treated as no label — no text node emitted, no exception. Interior empty/whitespace runs are normalised deterministically rather than drawn as gaps.
- **Overlong / oversized run**: each run is fitted to the region (wrap/shrink/truncate) measured in its own style, never overflowing horizontally or clipping mid-glyph; a single bold/large run cannot push the block past the region.
- **Too much styled content**: the drawn line count is capped to what the region holds; surplus is indicated (e.g. trailing ellipsis on the last drawn line) rather than overflowing vertically into the sigil/health/other channels.
- **Mixed sizes/weights on a line**: each line's height follows its tallest run and runs share a common baseline; mixing styles never causes vertical overlap or clipping.
- **Degenerate token (`R <= 0`) with a styled label**: the existing visible-placeholder rule applies and takes precedence; never a blank scene, never an exception.
- **No real measurer installed (pure-fallback path)**: the symbol's scene is still produced deterministically and the label still emits its per-run text nodes with their styles; tofu-free *rendering* and *wrapping at measured boundaries* are properties of the render edge (the render bridge installs the real measurer/shaper). The feature MUST NOT make the pure library depend on a measurer being installed, and MUST NOT throw when none is.
- **Determinism vs. measurement provider**: for a fixed measurement provider, identical styled-run-labelled `Token` ⇒ byte-identical scene. Where per-run wrapping and block geometry are computed from measured text, the result is a deterministic function of the resolved measurement; the contract is reproducibility under a fixed provider, not identical bytes across different providers.
- **Run colour vs. pre-attentive channels**: run colours are author-chosen, but a run coloured to mimic the faction or state channel encoding can mislead. The label remains an inspection-detail channel; avoiding such collisions is a design-loop guidance caveat (skill doc), not a silently enforced runtime rule, and it does NOT change the linter's pre-attentive governance.
- **Long Unicode / non-Latin runs**: each run is treated as an opaque short string in its own style; wrapping, fitting, and tofu-free behaviour follow whatever the installed real measurer/shaper supports (no new shaping/bidi responsibility is taken on by this feature).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The existing optional identity-label channel MUST support **styled runs** — a short, ordered sequence of text spans where each span ("run") may carry its own **colour, weight, and size** — carried through the **same shared channel** so a single `'stats -> Token` mapping drives every grammar unchanged; no new per-grammar mapping and no second label channel.
- **FR-002**: Rich-text styling MUST be **opt-in and layered-additive**: a symbol with **no** label MUST render **byte-identically** to the current (pre-feature) symbol, and a symbol whose label is **plain unstyled text** (single- or multi-line) MUST render **byte-identically** to spec 197 for the same channels (zero behavioural drift for no-label and unstyled content). A label of exactly one run in the default label style MUST be byte-identical to the equivalent plain label.
- **FR-003**: The set of per-run style attributes for this feature is exactly **colour, weight, and size**. Any run attribute MAY be omitted, in which case the run inherits the existing default label style for that attribute (so an all-default run reproduces the spec-197 uniform style). Other typographic attributes (italic, underline, strike, letter-spacing, per-glyph styling) are **out of scope** (FR-018).
- **FR-004**: Each grammar (Token, Badge, Ring) MUST draw the styled runs in the **same per-grammar label region** already used for the uniform label, in reading order, such that — when styling is present — the runs observably alter the output and do **not overlap** the sigil, health, or other channels.
- **FR-005**: When the symbol is rendered through the headless render bridge (which installs the real text measurer/shaper), **every run** of the label MUST be drawn with **real glyphs — tofu-free** — in its own style, and legible at the target on-board size. *Legible* is operationalized as **tofu-free** (`Missing = false` for every covered glyph of every drawn run) at the run's drawn size, with the design-loop eyeball pass as the human legibility check; the automated assertion is the tofu-free property, not a measured contrast/size threshold.
- **FR-006**: The styled label MUST be **fitted** to its region using real text measurement applied **per run in that run's own style**: runs MUST flow/wrap at measured boundaries (a run MAY break across lines), each line's height MUST follow its tallest run with runs sharing a common baseline, the number of drawn lines MUST be **capped** to what the region holds, surplus content MUST degrade gracefully (e.g. truncation with an ellipsis on the last drawn line), and the drawn block MUST NOT clip mid-glyph and MUST NOT overflow the footprint horizontally or vertically into adjacent channels.
- **FR-007**: A label that is **empty, all-whitespace, all-empty-runs, or otherwise reduces to no drawable glyphs** MUST be treated as no label (no text node, equivalent output to having no label) and MUST NOT raise; interior empty/whitespace runs MUST be normalised deterministically rather than drawn as wasted gaps.
- **FR-008**: A **degenerate token** (`R <= 0`) carrying a styled-run label MUST still degrade to the existing **visible placeholder** and MUST NOT raise; the placeholder rule takes precedence over the label.
- **FR-009**: The styled label MUST be **pure and deterministic** with respect to a fixed text-measurement provider: identical styled-run-labelled `Token` ⇒ identical scene ⇒ identical canonical bytes, in-process and across processes, with no wall-clock, randomness, or IO performed by the library. Per-run wrapping and block layout MUST be a deterministic function of the resolved measurement.
- **FR-010**: The pure symbology library MUST **NOT depend** on a real measurer being installed: it MUST emit a deterministic scene (including each run's styled text node) on the pure-fallback path and MUST NOT throw when no measurer is present. Tofu-free *rendering* and measured wrapping are properties of the render edge, not preconditions of the pure library.
- **FR-011**: A designer MUST be able to assemble a **review board** (gallery, and filmstrip where motion applies) of a styled-label roster in a selected grammar, reproducibly under a fixed measurement provider, with **no signature change** to the existing board/motion entry points (`render`, `galleryIn`, `filmstripIn`, `animateIn`).
- **FR-012**: The existing **legibility linter** (spec 194) MUST continue to return a **grammar-independent** verdict; if it reports the label at all, the label — plain or styled — MUST be treated as an **inspection-detail** channel that does not change the governance of the existing pre-attentive channels.
- **FR-013**: Run colours MUST be author-supplied from the existing scene `Color` vocabulary; the feature MUST NOT silently re-map or reject them at runtime. The guidance that run colours should not impersonate the faction/state pre-attentive encodings is a **design-loop / skill caveat**, not an enforced runtime rule, and MUST NOT change the linter's pre-attentive governance (consistent with the symbology colour policy that the **state/faction palettes** remain the pre-attentive channels — the label is inspection-detail).
- **FR-014**: The existing **Directional Token, Badge, and Ring** grammars and their motion/gallery/filmstrip behaviour MUST be **unchanged** for any `Token` without a label **and** for any `Token` whose label is plain unstyled text (zero behavioural drift); the existing channel set is otherwise untouched.
- **FR-015**: This is an additive change to the symbology package. If it alters the package's curated public surface (e.g. a new styled-run/label type), that surface baseline MUST be regenerated to capture the change, with **zero surface drift** on every other package baseline; if the styled capability reuses the existing label surface without adding new public surface, the baselines MUST remain unchanged and that MUST be recorded. Per Constitution II, any new public type MUST be declared in the module's `.fsi`.
- **FR-016**: The new styled capability MUST stay in the **pure scene-only layer** for scene construction — it adds no rendering, raster, GL, or IO dependency to the pure library; it depends only on the existing scene text/measurement vocabulary already available to the grammars (`measureTextResolved` / `glyphRunProof` / `FontSpec` / `Color`), and bundles **no new font files**.
- **FR-017**: The `fs-gg-symbology` design-loop skill MUST document rich-text runs — when to use them, the supported run attributes (colour / weight / size), that they require the real measurer for tofu-free output, that authors should keep to a few short runs and a restrained style/colour set, that run colours must not impersonate faction/state encodings, how surplus runs/width degrade, and that they complement rather than replace the vector sigil — authored canonically and mirrored to every skill tree, passing the skill-parity check.
- **FR-018**: The following remain **out of scope**: full rich-text layout (justification, alignment beyond the existing region siting, inline images, hyperlinks, lists), typographic attributes beyond colour/weight/size (italic, underline, strike-through, letter-/word-spacing, per-glyph styling), automatic generation or styling of labels from stats without a human in the loop, label-bound motion / animating runs, advanced bidirectional or complex-script typography beyond what the installed measurer already supports, any new GPU/compute path, and shipping new font files. The feature uses whatever font/measurer the render edge already provides.

### Key Entities *(include if feature involves data)*

- **Token (existing, extended)**: the fixed channel-set value describing one unit's encoded state. Its existing optional identity-label channel is **enriched** so the label may carry styled runs; all other channels and their meanings are unchanged. A `Token` with no label, or with a plain unstyled label, behaves exactly as before this feature.
- **Styled label run (new channel detail)**: one span of label text plus its optional per-run **colour, weight, and size**; omitted attributes inherit the default label style. An ordered sequence of runs forms the label's content. Inspection-detail, distinct from and complementary to the vector sigil; sited per grammar so the runs never collide with other channels; tofu-free when rendered through the real measurer.
- **Grammar (existing)**: the selectable form factor (Token / Badge / Ring). Each sites the label (now possibly styled) in its own region; the label is part of the one shared vocabulary, so the choice of grammar changes only where/how the runs are drawn, never the `ChannelMap`.
- **Text measurement provider (existing, at the render edge)**: the real measurer/shaper installed by the rendering edge that makes label glyphs tofu-free and drives **per-run** measurement, wrapping, and block fit. The pure library does not require it; the render bridge supplies it.
- **Legibility report (existing, spec 194)**: the governance artifact; grammar-independent, and unchanged in its pre-attentive-channel governance by the addition of the inspection-only label, whether plain or styled.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A roster's `'stats -> Token` mapping can attach a label of **styled runs** (per-run colour/weight/size) that renders in **all three** grammars with **zero** additional per-grammar mapping and **zero** signature change to the existing board/motion entry points.
- **SC-002**: A styled-run-labelled symbol rendered through the render bridge shows **every run** with **real glyphs (tofu-free)** in its own colour/weight/size; two labels with the **same characters** but **different run styling** produce **observably distinct** rendered output. *Distinguishable* and *legible* are operationalized as **tofu-free glyph runs** (`Missing = false`) on every drawn run plus **observably distinct rendered output** for distinct styling — not a measured contrast/size metric — with the eyeball-loop as the human check.
- **SC-003**: A `Token` with **no** label renders **byte-identically** to the current pre-feature symbol, and a `Token` with a **plain unstyled** label (single- or multi-line, including a single default-styled run) renders **byte-identically** to spec 197, in every grammar (layered zero behavioural drift).
- **SC-004**: For a fixed measurement provider, rendering the same styled-run-labelled `Token` twice — same process and separate process — yields **byte-identical** scene data (100% reproducible).
- **SC-005**: An overlong / oversized / over-numerous styled label stays within the symbol footprint with **zero** mid-glyph clips, **zero** horizontal or vertical overflow into adjacent channels, mixed-size lines sized to their tallest run, and a **bounded** drawn line count; an empty/whitespace/empty-run label and a degenerate-token-with-label both produce **zero** exceptions.
- **SC-006**: The legibility linter's verdict for a fixed roster is **identical** regardless of grammar, and is unchanged in its governance of the existing pre-attentive channels by the presence of plain or styled labels.
- **SC-007**: Surface baselines reflect the change exactly — only the symbology package's baseline moves if new public surface is added (else all baselines are unchanged and that is recorded), with **zero drift** on all other package baselines; the design-loop skill documents the rich-text run channel and passes the skill-parity check with **zero** critical/high findings.

## Assumptions

- **How styling is expressed**: the label is carried by the **same shared label channel** specs 196/197 introduced (the natural extension of the "one channel vocabulary" principle), now permitted to contain a short ordered sequence of **styled runs**. The exact field shape (e.g. widening `Token.Label` from `string option` to an optional list of styled runs, a parallel optional rich field, or a markup convention), the run-style record shape, the per-run measurement/wrap policy, and how a plain string maps to a single default-styled run are **implementation / design-loop details resolved at planning**, not contracts; the binding contracts are FR-002 (layered opt-in, zero drift for no-label and unstyled), FR-003 (attribute set = colour/weight/size), FR-004 (sited, observable, non-overlapping), FR-006 (per-run fitted, capped, no clip/overflow), FR-009 (deterministic), and FR-013 (author colours, guidance-only governance).
- **Run attributes**: exactly **colour, weight, and size**, each optional and inheriting the existing default label style when omitted (so the unstyled path is recovered exactly). This matches the deferred item as scoped in 197 FR-016 ("per-run colour/weight/size runs within the label"). Italic/underline/strike and other attributes are deferred (FR-018) to keep the channel a bounded widening rather than a layout engine.
- **What the label encodes**: a short, human-readable identity with an **emphasis hierarchy** over a few short runs (e.g. a loud callsign + a quiet code). It is an **inspection-detail** channel, not a pop-out one — the design-loop guidance keeps it to a few short runs with a restrained palette and treats the label as complementary to, not a replacement for, the vector sigil and the pre-attentive channels.
- **Tofu-free and measured wrapping are render-edge properties**: the pure library emits a deterministic scene with per-run styled text nodes; **legible, tofu-free glyphs** and **wrapping at measured boundaries** come from the real measurer/shaper that the render bridge already installs. The pure library neither installs nor requires a measurer and never throws without one — matching the existing seam where the rendering edge owns the real measurer.
- **Determinism is provider-relative**: identical styled-run-labelled `Token` ⇒ byte-identical scene **under a fixed measurement provider**. Where per-run wrapping and block geometry are computed from measured text, the output is a deterministic function of the resolved measurement; the feature does not promise identical bytes across *different* providers, only reproducibility under a fixed one.
- **Colour policy**: run colours are drawn from the existing scene `Color` vocabulary and are author-chosen; the symbology colour policy's pre-attentive channels remain the **faction/state palettes**. The label stays inspection-detail and the linter's governance is unchanged; "don't impersonate the pre-attentive encodings" is a documented design-loop caveat, not an enforced runtime constraint (FR-013).
- **Reused, unchanged**: the three grammars, the label channel and its per-grammar regions / wrap-cap-placeholder behaviour (specs 196/197), the legibility linter (spec 194), the headless render bridge (which installs the real measurer), the scene text/measurement vocabulary (`measureTextResolved`/`glyphRunProof`/`FontSpec`/`Color`), the board/motion entry points, and the design-loop skill scaffolding all exist and are reused as-is; this feature adds only the per-run styling of the label channel, its per-run measurement/siting, tests, any regenerated surface baseline, and the skill's rich-text documentation.
- **One vocabulary, one enriched channel**: no new grammars, no new label channel, no auto-generation, and no change to the channel-capacity governance beyond noting the label as inspection-detail — run styling as an enrichment of a single opt-in channel, not a new subsystem.
