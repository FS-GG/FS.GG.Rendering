# Quickstart: Validate the Showcase Control Fixes

A validation/run guide. Implementation detail lives in `tasks.md` (generated by
`/speckit-tasks`) and the implementation phase. References: [scroll-interaction](contracts/scroll-interaction.md),
[interaction-state](contracts/interaction-state.md), [control-pass-coverage](contracts/control-pass-coverage.md),
[data-model](data-model.md).

## Prerequisites

- .NET `net10.0` SDK; repository builds with standard tooling.
- For accepted live + visual evidence: a visible, focusable Linux desktop session with a working
  OpenGL/Skia presentation boundary. Headless lanes produce `environment-limited` results.

## Build and test

```bash
# Build the affected shared packages and the sample
dotnet build src/Controls src/Controls.Elmish src/SkiaViewer
dotnet build samples/SecondAntShowcase

# Deterministic regression (failing-first, then green after implementation)
dotnet test tests/Controls.Tests        # scroll offset/clamp/thumb, offset-aware hit-test, visual-state stamping
dotnet test tests/Elmish.Tests          # hover/focus/scroll routing + retained repaint parity
dotnet test tests/SkiaViewer.Tests      # wheel/scroll delivery
dotnet test samples/SecondAntShowcase/SecondAntShowcase.Tests   # per-control live-vs-scripted parity, classification, visual parity
```

## Scenario 1 — Scroll the content region (SC-001)

1. Run the sample interactively; open a page taller than the content region (e.g. a dense catalog
   page).
2. Drag the scrollbar thumb, use the wheel, and use scroll keys.
3. **Expected**: content scrolls, clamps cleanly at top and bottom, the thumb tracks the offset,
   and previously hidden bottom content becomes reachable. On a page that fits, no draggable thumb
   is shown.

## Scenario 2 — Hover and focus feedback (SC-002)

1. Move the pointer across each interactive control; tab focus through the page.
2. **Expected**: a visible hover state on pointer-over that clears on leave; a distinct focus
   affordance that moves with focus; combined hover+focus shows both; display-only controls show no
   interactive affordance.

## Scenario 3 — Every interactive control responds (SC-003, SC-004, SC-007)

1. For each interaction family, perform the contract's primary interaction with real input.
2. **Expected**: the contract's expected state change and visible evidence appear; the same result
   occurs whether driven by hand or by scripted coverage; display-only controls stay static.
3. Run the coverage check: every control classified, none failing.

## Scenario 4 — Appearance fidelity (SC-006)

1. Produce the visual review set for affected pages in antLight and antDark.
2. **Expected**: hover/focus/active/scroll affordances use the correct Ant palette roles in each
   appearance; no new spacing/alignment/clipping/contrast regression.

## Evidence to capture

- `readiness/finding-log.md` — every finding `ReVerified` (SC-005).
- `readiness/coverage-classification.md` — every control classified (SC-007).
- `readiness/responsiveness/` — live-responsiveness summary or `environment-limited` result; the
  Feature 174 budgets are not regressed.
- `readiness/visual-parity/` — light/dark review sets for affected pages.
- `readiness/validation-summary.md` — overall pass/limitation summary.

Disclose any synthetic substitute at the use site (`// SYNTHETIC: ...`), in the test name
(`Synthetic` token), and in the PR description.
