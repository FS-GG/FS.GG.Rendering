# Data Model: Shared Visual Readiness Tooling

## Visual Page

Sample-owned screen participating in a readiness run.

- `PageId`: stable non-empty id, unique within the run.
- `Title`: reviewer-facing label.
- `Order`: deterministic ordering key.
- `Required`: whether the page participates in readiness decisions.

Validation rules:

- `PageId` must be non-empty and unique.
- Ordering must be stable across runs for the same declaration.
- The shared package never renders a page; it only carries ids and labels.

## Visual Theme

Sample-owned visual mode participating in a readiness run.

- `ThemeId`: stable non-empty id, unique within the run.
- `Title`: reviewer-facing label.
- `Order`: deterministic ordering key.

Validation rules:

- `ThemeId` must be non-empty and unique.
- Theme aliases such as `antLight` and `antDark` remain sample-owned.

## Visual Size

Accepted viewport or render size required for visual readiness.

- `Width`: positive pixel width.
- `Height`: positive pixel height.
- `Role`: stable role such as `preferred`, `minimum`, or sample-defined text.
- `Order`: deterministic ordering key.

Validation rules:

- Width and height must be positive.
- Role must be non-empty when more than one size participates in a run.

## Capture Target

One required page, theme, size, and output path combination.

- `TargetId`: stable id derived from page, theme, size role/dimensions, and relative path.
- `Page`: `VisualPage`.
- `Theme`: `VisualTheme`.
- `Size`: `VisualSize`.
- `RelativePath`: path below the evidence root.
- `Required`: whether this target blocks readiness.

Validation rules:

- `TargetId` and `RelativePath` must be unique in a run.
- Paths must be relative to the evidence root and must not escape it.
- Duplicate or ambiguous targets fail before capture or validation begins.

## Capture Artifact

Observed filesystem evidence for a capture target.

- `RelativePath`: target-relative path.
- `Exists`: whether the file exists.
- `ByteCount`: file length when available.
- `DecodedWidth`: decoded width when available.
- `DecodedHeight`: decoded height when available.
- `ContentHash`: stable content identity when available.
- `DecodeError`: diagnostic when PNG decoding fails.

Validation rules:

- Missing files produce `missing`.
- Zero-byte, corrupt, or undecodable files produce `undecodable`.
- Decoded dimensions must match target size exactly unless the run records an accepted exception.
- Content hash uses SHA-256 over file bytes when the file is readable.

## Capture Record

Validation result for one capture target.

- `Target`: `CaptureTarget`.
- `Status`: `complete`, `missing`, `wrong-size`, `undecodable`, `degraded`, or `blocked`.
- `Artifact`: `CaptureArtifact option`.
- `ExpectedWidth` / `ExpectedHeight`.
- `ObservedWidth` / `ObservedHeight`.
- `Reason`: required for degraded and blocked statuses.
- `Diagnostics`: actionable messages.

Validation rules:

- `complete` requires a readable artifact, matching dimensions, and no blocking diagnostic.
- `degraded` requires a non-empty reason and is never accepted as complete.
- `blocked` requires an actionable reason.

State transitions:

```text
declared -> pending-capture -> complete
declared -> pending-capture -> missing
declared -> pending-capture -> wrong-size
declared -> pending-capture -> undecodable
declared -> pending-capture -> degraded
declared -> pending-capture -> blocked
```

## Reviewer Classification

Human review record for one required capture target.

- `TargetId`: target being reviewed.
- `Severity`: `none`, `minor`, `major`, or `blocking`.
- `DefectClass`: stable defect category or `none`.
- `ReadinessImpact`: reviewer-visible impact token.
- `Reviewer`: reviewer id or name.
- `ReviewedAt`: timestamp text or parsed timestamp when available.
- `Notes`: reviewer notes.

Validation rules:

- Every required target needs exactly one reviewer classification before acceptance.
- Unknown targets, duplicate targets, malformed rows, and pending rows are diagnostics.
- Any `blocking` severity blocks accepted readiness.
- `minor` and `major` do not block by themselves but remain visible in summaries.

State transitions:

```text
template-pending -> classified-none -> accepted-by-review
template-pending -> classified-minor -> accepted-with-note
template-pending -> classified-major -> accepted-with-note
template-pending -> classified-blocking -> blocked
template-pending -> missing-or-malformed -> pending-review
```

## Contact Sheet

Reviewer-facing visual index generated by a sample/app adapter.

- `SheetId`: stable id.
- `RelativePath`: path below evidence root.
- `ThemeId`: optional grouping key.
- `SizeRole`: optional grouping key.
- `TargetIds`: ordered targets included on the sheet.
- `MissingTargetIds`: targets omitted because artifacts were missing or undecodable.
- `Diagnostics`: generation notes.

Validation rules:

- Ordering must match target matrix order.
- Missing or undecodable targets must be listed rather than silently omitted.
- The shared package records and reports contact sheet metadata; AntShowcase owns PNG composition for this feature.

## Visual Readiness Report

Combined machine-checkable readiness record.

- `RunId`: stable run id.
- `EvidenceRoot`: root path used for validation.
- `Targets`: capture target list.
- `Captures`: capture records.
- `ReviewerClassifications`: parsed reviewer records.
- `ContactSheets`: contact sheet metadata.
- `StatusCounts`: counts by capture and reviewer status.
- `ReadinessStatus`: `accepted`, `pending-review`, `blocked`, `environment-limited`, or `incomplete`.
- `Caveats`: explicit limitations and accepted exceptions.
- `Diagnostics`: actionable messages.

Validation rules:

- Accepted readiness requires all required captures complete, all required reviewer classifications present, and no blocking reviewer defect.
- Environment-limited readiness is allowed only when captures are explicitly degraded with reasons; it is not accepted.
- Machine-readable report fields must be enough for validation lanes to avoid parsing prose.

## Generated Summary Section

Bounded generated content in a human-authored readiness summary.

- `StartMarker`: exact start marker.
- `EndMarker`: exact end marker.
- `Body`: generated Markdown.
- `InsertionPolicy`: deterministic insertion rule when markers are absent.
- `Diagnostics`: unsafe-write messages.

Validation rules:

- Exactly one start marker and one end marker are required for update.
- Missing markers are inserted in a predictable location.
- Multiple, reversed, or incomplete markers fail without writing.
- Manual content outside the generated section remains byte-for-byte unchanged.
